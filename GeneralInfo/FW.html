What a firewall is and how it began:

A firewall functions as a dedicated guardian positioned between two network zones, usually a trusted internal environment and the open internet, making per-packet decisions about which traffic is allowed through and which is denied. The concept surfaced in 1988 at Digital Equipment Corporation (DEC), where engineers built the first packet-filter firewall to curb a wave of early-internet worms. Researchers at AT&T Bell Labs and elsewhere soon advanced the idea into application proxy firewalls that could interpret higher-level protocols. By 1993-1994, Check Point introduced stateful inspection, enabling devices to follow the entire life cycle of a connection instead of treating each packet in isolation. When rising bandwidth and more sophisticated attacks defined the 2000s, vendors merged multiple functions such as, IPS, URL filtering, antivirus and many more into a single platform known as a unified threat management (UTM) firewall. Another milestone arrived in 2009, when Palo Alto Networks launched the first widely adopted next-generation firewall (NGFW), which added application awareness, user identity mapping, and cloud-based threat intelligence. Today, “firewall” can describe a hardware appliance, virtual machine, cloud-native service, or lightweight host agent, but the mission remains the same: enforce security policy at a trust boundary.


What a firewall actually does:

Modern firewalls perform several intertwined jobs:

  • Traffic classification and filtering: Inspect packet headers and, if required, payloads to enforce allow/deny rules based on IP addresses, ports, protocols, users, applications, or content signatures.
  • State tracking: Maintain a per-flow table so return traffic is recognised as legitimate and not re-evaluated from scratch, boosting both performance and security.
  • Network address translation (NAT): Hide internal addressing schemes and conserve public IP space
  • Segmentation: Logically separate workloads or departments to reduce lateral-movement risk.
  • Advanced threat prevention: Inline intrusion-prevention engines, TLS/SSL inspection, sandboxing, and machine-learning feeds detect evasive malware and command-and-control callbacks.
  • Visibility and logging: Generate high-fidelity logs, flow records, and dashboards for audit, forensics, and compliance.
  • Quality of Service (QoS): Prioritizes latency-sensitive applications like VoIP during network congestion.
  • VPN Integration: Secures remote access through encrypted tunnels while maintaining firewall inspection capabilities.
In short: It filters the good from the bad, keeping the nasties out and sometimes preventing your infected device from causing trouble elsewhere.

Host-based vs Network-based firewalls:

  • Host-Based Firewall:

    • Host-based firewalls (HBFs) constitute software applications installed directly on endpoint devices, computer, servers, workstations, or mobile devices, to monitor and control their specific network traffic. Unlike perimeter focused solutions, HBFs provide individualized security policies tailored to each host's operational requirements. These firewalls intercept all incoming/outgoing communications at the operating system level, applying rules based on application binaries, user identities, and process behaviors. For example, an HBF might permit a specific accounting software to communicate only with designated internal servers while blocking all other outbound connections, thereby containing potential malware exfiltration.

    Pros:

    • Personalized protection tailored to the device.
    • Protects you even when you're on untrusted networks (traveling, public Wi-Fi).
    • Can control traffic between applications on the same device.
    • Usually easier for individuals to configure (Ex: Windows Defender Firewall settings).

    Cons:

    • Only protects the device it's installed on.
    • Relies on the device's own resources (CPU, memory).
    • Needs to be managed on each device individually.
    • Examples: Windows Defender Firewall (built into Windows), macOS Application Firewall, third-party software like Norton or McAfee personal firewalls.


  • Network-Based Firewall:

    • Network-based firewalls (NBFs) operate at network boundaries, typically between internal LANs and external WANs, as dedicated hardware appliances or virtual instances. These systems inspect all traffic traversing network segments, enforcing centralized security policies that protect entire subnets. NBFs employ high-throughput packet processing engines capable of handling multi-gigabit traffic volumes, utilizing specialized hardware like ASICs for acceleration. Deployment scenarios include internet gateways, filtering external traffic, internal segmentation, controlling inter-VLAN communication, and cloud perimeters, securing virtual networks in IaaS environments.

    Pros:

    • Protects every device on the internal network behind it with one system.
    • Offloads security processing from individual devices.
    • Centralized management and logging for the entire network.
    • Often much more powerful and feature-rich than host-based firewalls.

    Cons:

    • Doesn't protect devices if they roam outside the network (Ex: your laptop at a cafe).
    • Doesn't control traffic between devices already inside the network.
    • Usually requires more technical skill to set up and manage properly.
    • Examples: The firewall built into your home Wi-Fi router, dedicated enterprise firewall appliances from Cisco, Fortinet, Palo Alto Networks, or software firewalls like pfSense/OPNsense running on dedicated hardware.


Core network-based firewall:

  • Packet-Filtering Firewalls (stateless): Represent the foundational architecture in firewall technology, operating at the network layer (Layer 3) and transport layer (Layer 4) of the OSI model. These systems inspect individual data packets in isolation, applying allow/deny decisions based on predefined rules that evaluate packet headers. Key evaluation criteria include source/destination IP addresses, source/destination port numbers, and communication protocols (TCP, UDP, ICMP).

    • Advantage:

    • High Performance and Low Latency: By examining only packet headers rather than payload content, packet-filtering firewalls achieve exceptionally fast processing speeds, often handling multi-gigabit traffic volumes with minimal latency. This efficiency makes them ideal for high-throughput environments like internet service providers or backbone networks.
    • Resource Efficiency: Minimal computational requirements allow deployment on low-cost hardware, including routers with integrated firewall capabilities.
    • Transparent Operation: Unlike proxy-based solutions, packet filters don't require client reconfiguration since they operate transparently at the network layer.
    • Simplified Rule Verification: Administrators can easily test filtering rules due to straightforward allow/deny conditions based on header parameters.
    • Cheap/Easy to Implement: Often built into basic network hardware.

    Disadvantage:

    • Limited Context Awareness: Inability to track connection states leaves these firewalls vulnerable to IP spoofing attacks and advanced evasion techniques where malicious packets mimic legitimate header information.
    • No Application-Layer Inspection: Lack of payload analysis prevents detection of application-layer threats like SQL injection or malware embedded within allowed protocols.
    • Static Rule Limitations: Rule sets become cumbersome in complex environments, often leading to configuration errors or unintended access loopholes.
    • Inadequate for Modern Threats: Cannot defend against sophisticated attacks leveraging allowed ports/protocols for command-and-control communications or data exfiltration.
Ideal for network segments requiring basic access control without deep inspection, such as internal network segmentation between departments or as a first-line filter before more advanced firewalls.
  • Circuit-level gateway: Circuit-level gateways (CLGs) operate at the session layer (Layer 5) of the OSI model, focusing exclusively on validating the initiation of network sessions rather than inspecting packet contents. These systems establish virtual "circuits" between internal clients and external servers, acting as intermediaries that authenticate session requests before permitting data flow. Once a session is approved, packets flow directly without further inspection

    • Advantage:

    • Session Validation: Prevents unauthorized session establishment a critical defense against reconnaissance scans and connection hijacking attempts.
    • Network Obfuscation: Hides internal IP addresses and topology details from external networks.
    • High Throughput: Minimal processing overhead after session establishment enables near-line-speed performance.
    • Protocol Agnosticism: Works with any TCP/UDP-based protocol without requiring protocol-specific proxies.

    Disadvantage:

    • No Content Inspection: Inability to examine packet payloads leaves systems vulnerable to malware, data exfiltration, and application-layer attacks transmitted within allowed sessions.
    • Limited Attack Protection: Cannot detect or prevent attacks leveraging established sessions (Ex: authenticated users executing malicious commands).
    • No User Authentication: Lacks integrated user authentication mechanisms, relying solely on IP/port validation.
    • Stateful Limitations: Basic implementations may struggle with complex protocols involving dynamic port assignments (Ex: FTP passive mode).
Effective as a lightweight supplement to other firewalls in environments requiring fast session validation such as high-traffic web server farms or as a component in VPN gateways.
  • Stateful inspection: Stateful inspection firewalls enhance traditional packet filtering by maintaining dynamic context about active network connections. Operating primarily at the transport layer (Layer 4), these systems track the state of TCP/UDP sessions including sequence numbers, connection handshakes, and session duration to validate packet legitimacy. For instance, they distinguish legitimate reply packets in an established FTP session from unsolicited inbound connection attempts

    • Advantage:

    • Contextual Decision-Making: By understanding connection states, these firewalls prevent IP spoofing and certain denial-of-service attacks that exploit stateless architectures.
    • Protocol Compliance Enforcement: Can enforce proper protocol behavior (Ex: ensuring TCP handshake completion) to block malformed packet attacks.
    • Dynamic Rule Application: Handles dynamic ports (ex: passive FTP) automatically.
    • Enhanced Logging: Detailed session records provide richer forensic data compared to packet filters4.

    Disadvantage:

    • Resource Intensive: Connection state tables consume significant memory and processing resources, potentially limiting scalability in high-connection environments.
    • Application-Layer Blindness: Like packet filters, they cannot inspect payload content for application-layer threats unless integrated with deeper inspection capabilities.
    • Vulnerability to Flooding Attacks: State table exhaustion attacks can crash the firewall by overwhelming it with fake connection requests.
    • Complex Configuration: Rule sets must account for state dependencies, increasing administrative overhead.
Well-suited for enterprise network perimeters where balancing security and performance is critical, particularly for environments with predictable traffic patterns.
  • Next-generation firewall (NGFW): Next-generation firewalls integrate traditional stateful inspection with advanced security services, including deep packet inspection (DPI), intrusion prevention systems (IPS), application awareness, and threat intelligence feeds. Operating across Layers 2 to 7, NGFWs identify applications regardless of port/protocol (Ex: detecting Facebook traffic on non-standard ports) and enforce user/group-based policies.

    • Advantage:

    • Application Visibility and Control: Identifies thousands of applications (including encrypted traffic) to enforce granular policies (Ex: "block Dropbox but allow OneDrive").
    • Integrated Threat Prevention: Combines firewall, IPS, antivirus, and sandboxing to block known exploits, malware, and zero-day threats.
    • User-Centric Policies: Enables rules based on user identity (via LDAP/AD integration) rather than just IP addresses.
    • SSL/TLS Inspection: Decrypts and inspects encrypted traffic for hidden threats.

    Disadvantage:

    • High Cost: Subscription fees for threat intelligence and advanced features significantly increase total cost of ownership.
    • Performance Impact: Resource-intensive features like SSL inspection and DPI can introduce latency, especially on lower-end models.
    • Management Complexity: Unified security requires sophisticated configuration and ongoing tuning to balance security and usability.
    • Potential Overblocking: Overly aggressive application control or IPS rules may disrupt legitimate business applications.
Essential for modern networks requiring comprehensive protection against advanced threats particularly in environments with diverse applications, encrypted traffic, or regulatory compliance requirements.
  • Proxy firewalls: A proxy is a layer 7 firewall terminates the client’s session, inspects the full protocol dialogue, then opens a new session to the destination on the client’s behalf. Because each side talks only to the proxy, the original end-points never see one another. Here how it works?
    • Client → [Proxy Firewall] → Internet Server (Inspects HTTP/FTP/SMTP content)
    Step 1: Client sends request to the proxy (Ex: http://proxy:8080).
    Step 2: Proxy validates protocol compliance, inspects payload (Ex: checks for SQLi in HTTP POST).
    Step 3: If safe, proxy rebuilds the request from itself to the server.

    Advantage:

    • Ultra-Deep Inspection: Reads actual content (Ex: blocks malware in email attachments).
    • Protocol Sanitization: Fixes RFC violations or drops malformed packets.
    • Anonymity: Hides internal IPs; external servers only see the proxy.

    Disadvantage:

    • Scalability Issues: Struggles with high traffic volumes (Ex: 10Gbps+ networks).
    • Client Configuration: Often needs manual proxy settings on endpoints.

What is a firewall manager:

A firewall manager is a centralised policy-orchestration and operations hub. It:

  • Policy Orchestration: Push consistent rules to 20+ firewalls with one click.
  • Device Monitoring: Health checks, traffic analytics, threat dashboards.
  • Version Control: Track policy changes (who changed what, when).
  • Automation: Script mass updates (Ex: block a malicious IP globally).
  • Compliance Reporting: Generate PCI/HIPAA audit trails.
  • Log view: Log view of all traffic from a single place.
  • User Managment: Easy to manage users across firewalls and set roles to them.
  • Scalability: Centrally manages 500+ firewalls across hybrid environments (cloud/on-prem).
  • Admin Role Delegation: Restricts junior staff to view-only access while allowing policy edits by seniors.
  • Firmware Management: Synchronizes security updates across all devices.

Without a manager, large estates drift into inconsistent rules, mismatched firmware versions, and blind spots in compliance evidence.

Default-Deny vs Default-Allow Policies:

Policy Type Approach Advantages Disadvantages Best For
Default-Deny Blocks all traffic except explicitly allowed Most secure approach, minimizes attack surface Higher admin overhead, can break legitimate apps High-security environments, regulated industries
Default-Allow Permits all traffic except explicitly blocked Easier to implement, fewer user complaints Security risks from overlooked threats Internal test networks, legacy environments

Best-practice :

Drawing on my experience, I’ve assembled the following comprehensive security best-practice guide:

  • Lock down the management plane:
    • Place the MGMT interface on an out-of-band VLAN or loopback; no Internet-routable IP.
    • Whitelist source IPs for SSH/HTTPS/SNMP.
    • Enforce multi-factor auth and unique named accounts; disable default passwords, anonymous CLI, and legacy protocols such as Telnet, HTTP, SSH-v1, TLS < 1.2.
    • Disable USB management ports and shut down other unused ports.
    • Why it matters: Attackers usually target the control channel first; an isolated, MFA-gated management plane prevents an entire estate takeover.

  • Define a default-deny policy:
    • Create an explicit “deny all” rule at the end of every rulebase.
    • Build allow rules only for documented business flows; tag each with owner, ticket ID, expiration date.
    • Sort rules top-down by specificity: fixed IP/port entries first, broad wildcards last.
    • Why it matters: Eliminates shadow ports and reduces rule-shadowing. Any surprise traffic is logged, investigated, and either justified or blocked.

  • Enforce least-privilege user roles:
    • Use RBAC to carve roles such as Read-Only Ops, Log-Analyst, Policy Author, Firmware Maintainer.
    • Grant CLI “configure” or commit rights only to a narrow group, and require change-control tickets for elevation.
    • Enforce 90-day password rotation with MFA and SSH keys/certificates elsewhere.
    • Session timeouts (15 min inactivity).
    • Why it matters: Compartmentalises human error and insider risk; a log analyst cannot slip a backdoor rule into production.

  • Harden IPsec & remote-access VPNs:
    • Prefer IKEv2, AES-GCM, DH group 19+ (ECDH P-256) or group 20/21 for higher assurance.
    • Set IPSec Encryption to 256 and Authentication to SHA256.
    • Disable aggressive mode and weak hashes (MD5, SHA-1).
    • Pin peer certs to private CAs; if PSK must be used, rotate quarterly.
    • Apply split-tunnelling only for explicitly allowed subnets; otherwise force all traffic through VPN for inspection.
    • Why it matters: Prevents downgrade, MITM, and credential-stuffing attacks; isolates remote workers from lateral movement until posture checks pass.

  • Patch, back up, and test:
    • Track vendor advisories; schedule upgrades quarterly or immediately for critical CVEs.
    • Back up running config on every change commit; encrypt backups with the organisation’s key vault.
    • Replicate configs to a lab appliance; run automated regression tests (functional and performance) before production push.
    • Why it matters: Ensures vulnerabilities are closed fast without unexpected outage; rollback is one file away.

  • Automate compliance & change control:
    • Integrate the firewall manager (Panorama, FortiManager, Smart-1, etc.) with ITSM so no change is committed without a ticket reference.
    • Schedule nightly policy-drift reports that diff running vs golden config.
    • Enforce four-eyes approval on any rule wider than /24 or port 0-65535.
    • Why it matters: Cuts configuration drift and catches risky “Friday-night” changes before they propagate.

  • High Availability (HA):
    • Position firewalls in active-active or active-standby pairs and enable full state-table synchronization so established sessions survive a switchover.
    • Dual power feeds, dual WAN/LAN links on separate switch stacks, and dynamic routing (OSPF/BGP) or VRRP/HSRP make the data path as redundant as the appliances.
    • Use heartbeat links plus dataplane probes (e.g., upstream ping or synthetic app checks) to trigger failover only when both control and data functions degrade.
    • Why it matters: HA eliminates the firewall as a single point of failure, maintains session continuity during hardware faults or software upgrades, and upholds SLAs without exposing the network to an unprotected bypass path..

  • Logging & Monitoring:
    • Critical Alerts to Enable:Port scan detected, Admin login failures, VPN brute-force attempts...etc.
    • Forward logs to Splunk/QRadar or any SIEM solution for correlation.
    • Change heading to Full-fidelity logging & monitoring.
    • Capture session start/stop, NAT before/after.
